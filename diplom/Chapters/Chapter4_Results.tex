\section{Харагдах хэсгийн хөгжүүлэлтийн үр дүн (Frontend)}
\label{sec:frontend_results}

Flutter дээрх харагдах хэсэгт (frontend) дараах үндсэн дэлгэцүүдийг хэрэгжүүлж, туршилтаар баталгаажуулав.

\begin{itemize}
    \item Нэвтрэх/бүртгэх (Login/Signup): Нэвтрэх болон бүртгэх урсгал;
    \item Хянах самбар (Dashboard): Өнөөдрийн тэмдэглэл, сэтгэл хөдлөлийн сонголт (mood picker), түргэн үйлдлүүд (quick actions);
    \item Календарь (Calendar): Огноогоор шүүлттэй тэмдэглэлийн харагдац;
    \item Стикер (Sticker): Заавар текст (prompt) оруулж хиймэл оюун ухаанаар стикер үүсгэх интерфейс;
    \item Мэдэгдлүүд (Notifications): Хэрэглэгчийн интерфейсийн (UI) түвшний мэдэгдлийн жагсаалт;
    \item Хувийн хэсэг (Profile): Хэрэглэгчийн мэдээлэл, системээс гарах (logout).
\end{itemize}

Навигаци (navigation) нь доод таб бүтэцтэй (bottom tab), \texttt{IndexedStack} ашигласнаар дэлгэц хооронд шилжих үед төлөв хадгалах нөхцөл бүрдсэн.

\noindent \textbf{Тайлбар (Dashboard):}
Dashboard дэлгэц нь хэрэглэгчийн өдөр тутмын хэрэглээний гол төв болохыг зурагт харуулсан. Дээд хэсэгт хэрэглэгчийн мэндчилгээ, төлөвийн товч мэдээлэл байршиж, дунд хэсэгт mood сонголтууд болон \textit{Add Note} товчоор шинэ тэмдэглэл үүсгэх урсгал эхэлдэг. \textit{Today\'s Notes} болон \textit{Recent Notes} хэсэгт тухайн өдрийн болон сүүлийн тэмдэглэлүүдийг нэг дор харуулснаар CRUD үйлдэл рүү (засах/устгах) шууд шилжих боломж бүрдэж, хэрэглэгчийн үйлдлийн алхмыг цөөлсөн.

\noindent \textbf{Тайлбар (Calendar):}
Calendar дэлгэцийн зурагт огноонд суурилсан тэмдэглэлийн шүүлт хэрхэн ажиллаж байгааг харуулсан. Сарын харагдац дээр тэмдэглэлтэй өдрүүдийг визуал тэмдэглэгээгээр ялгасан бөгөөд тодорхой өдөр сонгоход доод хэсэгт \textit{Selected day notes} жагсаалт автоматаар шинэчлэгдэж байна. Энэ нь \texttt{noteDate} талбарт суурилсан өгөгдлийн шүүлт UI түвшинд зөв хэрэгжсэнийг баталгаажуулж, хэрэглэгчид өнгөрсөн тэмдэглэлээ хурдан харах боломж олгосон.

\noindent \textbf{Тайлбар (AI sticker generation):}
Sticker generation дэлгэц дээр хэрэглэгч mood болон prompt текстээ оруулж \textit{Generate} товчоор AI үйлчилгээ дуудсаны дараах үр дүнг шууд preview хэлбэрээр үзүүлж байгааг зурагт харуулсан. Мөн \textit{Calendar дээр хадгалах} товчоор үүссэн стикерийг тэмдэглэлийн урсгалтай уялдуулан хадгалах боломжтой бөгөөд ингэснээр AI-ээр үүсгэсэн контент нь зөвхөн зураг үүсгэх түвшинд зогсохгүй тэмдэглэл, календарь, сануулгын модультай интеграцчлагдсан нь батлагдсан.

\subsection{Мэдэгдлийн (notification) модулийн хэрэгжилт}
\label{subsec:notification_result}

Энэхүү системд мэдэгдлийн функцийг төхөөрөмжийн мэдэгдэл (notification) болон апп доторх мэдэгдлийн төв (notification center) гэсэн хоёр түвшинд хэрэгжүүлсэн. Аппликейшн эхлэх үед мэдэгдлийн үйлчилгээ инициализаци хийгдэж, Android/iOS орчинд шаардлагатай зөвшөөрлүүдийг (permission) хүсэх механизмыг ажиллуулдаг. Android талд \texttt{sticker\_events\_channel} суваг үүсгэснээр сануулга, стикертэй холбоотой event-үүдийг тусгай сувгаар хүргэх боломж бүрдсэн.

Мэдэгдлийн урсгалын хувьд дараах үндсэн зан төлөв баталгаажсан.

\begin{itemize}
    \item Стикер үүссэний дараах мэдэгдэл: стикер амжилттай үүсэхэд шууд мэдэгдэл үзүүлж, боломжтой үед зурагт суурилсан урьдчилан харах (big picture preview) хэлбэрээр харуулсан;
    \item Сануулга хуваарьлах (reminder scheduling): \texttt{reminderAt} талбарт суурилан тухайн тэмдэглэлд хамаарах мэдэгдлийг хугацаатайгаар хуваарьлаж, засвар/устгалын үед өмнөх хуваарийг цуцалж (cancel) дахин тохируулдаг;
    \item Мэдэгдлийн төв дэлгэц: \texttt{reminderAt} эсвэл \texttt{stickerUrl} бүхий тэмдэглэлүүдийг нэгтгэн мэдэгдлийн жагсаалт үүсгэж, уншсан/уншаагүй төлөв болон таб дээрх badge-тэй уялдуулсан.
\end{itemize}

Цагийн бүсийн (timezone) зөрүүгээс сэргийлэхийн тулд сануулгын хугацааг \texttt{TZDateTime.from(reminderAt.toUtc(), UTC)} хэлбэрээр хөрвүүлэн төлөвлөдөг шийдэл ашигласан.

\subsection{UI/UX туршлагын ажиглалт}
\label{subsec:uiux_observation}

Хэрэгжилтийн туршид дараах хэрэглэгчийн туршлагын (UX) шийдлүүд эерэг нөлөө үзүүлсэн.

\begin{itemize}
    \item Ачааллын (loading) төлөв бүрт тодорхой заагч (indicator) ашигласан;
    \item Алдааны мессежийг хэрэглэгчид ойлгомжтой байдлаар харуулсан;
    \item CRUD үйлдлийн дараа хэрэглэгчийн интерфейсийг автоматаар шинэчилж (refresh), өгөгдлийн зөрүүг бууруулсан;
    \item Сэтгэл хөдлөлийн төлөв (mood) болон дүрст тэмдэг (emoji) ашигласнаар тэмдэглэл оруулах оролцоо нэмэгдсэн.
\end{itemize}

\section{Сервер талын хөгжүүлэлтийн үр дүн (Backend)}
\label{sec:backend_results}

Node.js/Express дээрх сервер талын хэсэгт (backend) чиглүүлэлт (route) болон хянагч (controller)-ийг салгаж зохион байгуулсан нь системийн засварлах, өргөтгөх чадварыг (maintainability) сайжруулсан. Хэрэгжсэн боломжууд:

\begin{itemize}
    \item Баталгаажуулалтын API (Auth API): бүртгэл/нэвтрэлт (signup/login), bcrypt нууцлалын hash, JWT үүсгэлт;
    \item Тэмдэглэлийн API (Note API): үүсгэх, унших, шинэчлэх, устгах (create/read/update/delete);
    \item Эрхийн хяналт (Authorization): Bearer токен шалгах завсрын модуль (middleware);
    \item Алдааны боловсруулалт (error handling): токен, API түлхүүр, сүлжээний алдааг стандарт мессежээр буцаах.
\end{itemize}

MongoDB талд \texttt{User}, \texttt{Note} загваруудын талбарууд бодит хэрэглээний урсгалыг хангаж, нэмэлт атрибут өргөтгөх суурьтай болсон.

\section{Системийн баталгаажуулалт ба ерөнхий үнэлгээ}
\label{sec:system_validation}

Энэхүү судалгааны хүрээнд AI Notebook системийн frontend, backend, өгөгдлийн сан болон AI sticker generation интеграц бүрэлдэхүүнүүд хоорондын уялдаа ажиллах түвшинд хүрсэн. Үр дүнг системийн түвшинд авч үзвэл:

\begin{enumerate}
    \item Хэрэглэгчийн баталгаажуулалтын урсгал амжилттай;
    \item Тэмдэглэлийн амьдралын мөчлөг (үүсгэх-засах-устгах) тогтвортой;
    \item Календарь болон mood төлөвөөр тэмдэглэл харах боломж хэрэгжсэн;
    \item AI sticker generation төгсгөлийн цэг амжилттай интеграцчлагдсан;
    \item JWT middleware-аар хамгаалагдсан API зохих ёсоор ажилласан.
\end{enumerate}

\subsection{Дарааллын диаграммын түвшний баталгаажуулалт (Sequence)}

\fref{fig:seq_login}, \fref{fig:seq_note_create}, \fref{fig:seq_sticker} гэсэн дарааллын диаграммуудаар харуулсан хугацааны дарааллыг туршилтаар баталгаажуулав:

\begin{enumerate}
    \item \textbf{Нэвтрэх (login) урсгал}: POST /api/auth/login хүсэлт, bcrypt шалгалт, JWT үүсгэлт, Secure Storage-д хадгалалт;
    \item \textbf{Тэмдэглэл үүсгэх (note create) урсгал}: POST /api/notes хүсэлт, middleware-р JWT шалгах, MongoDB-д хадгалалт, UI refresh;
    \item \textbf{Стикер үүсгэх (sticker generate) урсгал}: POST /api/notes/generate-sticker хүсэлт, OpenRouter дуудлага, URL/алдааны хариу, preview харгалт.
\end{enumerate}

\section{Эрсдэл ба хязгаарлалтын үнэлгээ}
\label{sec:risk_and_limits}

Хэрэгжилтийн явцад илэрсэн гол хязгаарлалтууд:

\begin{enumerate}
    \item Сервер талд \texttt{MONGO\_URI} болон \texttt{JWT\_SECRET}-ийн анхдагч (fallback) утга ашиглаж байгаа нь production орчинд аюултай;
    \item \texttt{OPENROUTER\_API\_KEY}-д хамааралтай тул гадны үйлчилгээ тасалдах эрсдэлтэй;
    \item Мэдэгдлийн урсгал нь \texttt{/api/notes} өгөгдөл дээр клиент талд нэгтгэгдэж байгаа бөгөөд тусдаа мэдэгдлийн API болон FCM/APNs бүрэн нэгтгэгдээгүй;
    \item Token refresh механизм хийгдээгүй тул урт хугацааны session management сайжруулах шаардлагатай.
\end{enumerate}
